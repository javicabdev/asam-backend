# Migration 000003: Allow NULL member_id for Family Payments

## Overview

This migration modifies the `payments` table to allow `NULL` values for `member_id`, enabling family-only payments without requiring an associated member.

## Problem Solved

Previously, all payments required a `member_id` (NOT NULL constraint), which caused errors when attempting to register payments for families without specifying a member. This was incorrect because family payments should be associated only with a `family_id`.

## Changes

### Database Schema Changes

1. **Column Modification:**
   - `payments.member_id`: Changed from `NOT NULL` to `NULLABLE`

2. **New Constraint:**
   - Added CHECK constraint: `payments_member_or_family_check`
   - Ensures at least one of `member_id` OR `family_id` is present
   - Prevents invalid payments with neither association

### Code Changes

1. **Domain Model (`internal/domain/models/payment.go`):**
   - `Payment.MemberID`: Changed from `uint` to `*uint`
   - `Payment.Member`: Changed from `Member` to `*Member`
   - Updated `Validate()` method to handle nil pointers

2. **Service Layer (`internal/domain/services/payment_service.go`):**
   - Updated `validatePayment()` to check for nil before dereferencing
   - Updated `GetDefaulters()` to skip family-only payments
   - Updated `SendPaymentConfirmation()` to handle nil MemberID

3. **Resolver Layer (`internal/adapters/gql/resolvers/payment_resolver.go`):**
   - Updated `mapPaymentInputToModel()` to assign pointer
   - Updated `validatePaymentAssociations()` to check for nil

4. **CashFlow Model (`internal/domain/models/cashflow.go`):**
   - Updated `NewFromPayment()` to handle optional MemberID

## How to Apply

### Option 1: Using Docker Compose (Recommended for Development)

If you're running the application with Docker Compose, the migration will be applied automatically on the next restart:

```bash
# Stop the containers
docker-compose down

# Rebuild and start (migrations run automatically)
docker-compose up --build
```

### Option 2: Using Migration Scripts

**Linux/macOS:**
```bash
chmod +x scripts/db/apply_migration_003.sh
./scripts/db/apply_migration_003.sh
```

**Windows PowerShell:**
```powershell
.\scripts\db\apply_migration_003.ps1
```

### Option 3: Using Go Migrate Tool Directly

```bash
# Load environment variables
source .env.development  # or .env.aiven for production

# Apply migration
migrate -path ./migrations -database "$DATABASE_URL" up 1

# Check current version
migrate -path ./migrations -database "$DATABASE_URL" version
```

### Option 4: Manual SQL Execution

If you prefer to run the SQL manually:

```sql
-- Remove NOT NULL constraint from member_id column
ALTER TABLE payments ALTER COLUMN member_id DROP NOT NULL;

-- Add check constraint
ALTER TABLE payments 
    ADD CONSTRAINT payments_member_or_family_check 
    CHECK (member_id IS NOT NULL OR family_id IS NOT NULL);
```

## Rollback

To rollback this migration:

```bash
migrate -path ./migrations -database "$DATABASE_URL" down 1
```

**⚠️ WARNING:** Rolling back will DELETE all family-only payments (where `member_id` IS NULL) before restoring the NOT NULL constraint.

## Impact Analysis

### Breaking Changes
- ✅ **None** - All existing payments have a valid `member_id`, so no data migration needed
- ✅ **Backward Compatible** - Existing member payments continue to work as before

### New Functionality
- ✅ Family payments can now be registered without a `member_id`
- ✅ Database enforces business rule: at least one of `member_id` or `family_id` required

### Performance Impact
- ✅ **Minimal** - Added CHECK constraint has negligible performance impact
- ✅ Existing indexes on `member_id` remain functional

## Related Issues

This migration fixes the following error that occurred when registering family payments:

```
ERROR: insert or update on table "payments" violates foreign key constraint "payments_member_id_fkey" (SQLSTATE 23503)
```
